# VANTARIS IBMS PostgreSQL Target Schema Principles

## 1. Target DB Positioning

PostgreSQL is the canonical relational database for IBMS runtime configuration, users/roles/permissions, system menus, versions, asset topology, device metadata, integration metadata, ESG/MMS operational records, audit logs, and evidence indexes.

MySQL remains legacy runtime only until migration phases complete. **MySQL local-smoke is paused.**

---

## 2. Schema Design Principles

- Explicit primary keys (UUID `varchar(32)` hex or `bigint` identity — match existing IBMS conventions per table)
- **timestamptz** for event/audit timestamps (replace naive `DateTime` + `CURRENT_TIMESTAMP` where possible)
- **jsonb** for flexible payloads (`extra`, `connect_config`, `permission_codes`, VC JSON)
- UUID where global uniqueness is needed across services
- Foreign keys for core relationships (entity types, field mappings, version-menu links)
- Indexes for tenant/site/system/device/time/status query paths
- Soft delete only where product requires it (not assumed globally)
- Audit fields (`created_at`, `updated_at`) on core tables — implement `updated_at` via trigger or ORM `onupdate`
- No production migration without backup/rollback plan
- UFMS tables must not be merged into IBMS schema

## 5. Type Mapping (MySQL legacy → PostgreSQL target)

| Legacy (IBMS/MySQL) | PostgreSQL target |
| ------------------- | ------------------- |
| `BIGINT AUTO_INCREMENT` | `bigint GENERATED BY DEFAULT AS IDENTITY` |
| `CHAR(32)` UUID hex | `varchar(32)` or `uuid` (if normalized) |
| `JSON` | `jsonb` |
| `SMALLINT` 0/1 flags | `boolean` (preferred) or `smallint` |
| `DATETIME` | `timestamptz` |
| `DECIMAL(20,6)` | `numeric(20,6)` |
| `TINYINT` (IoT dialect) | `smallint` or `boolean` |
